# Отчёт по проекту DeepSort2

## 1. Базовая линия (оригинальный DeepSORT)

### 1.1. Условия эксперимента

- **Реализация**: оригинальный DeepSORT (nwojke/deep_sort), клонирован из `https://github.com/nwojke/deep_sort.git`
- **Параметры по умолчанию**:
  - `min_confidence` = 0.3
  - `nms_max_overlap` = 1.0
  - `min_detection_height` = 0
  - `max_cosine_distance` = 0.2
  - `nn_budget` = 100
- **Детекции**: предзагруженные `det.txt` из MOT Challenge (POI-детектор)
- **REID-фичи**: случайные 128-мерные векторы (оригинальная TensorFlow-модель `mars-small128.pb` недоступна)
- **Окружение**: Python 3.13.3, CPU (без GPU), numpy 1.26.4, scipy 1.17.1, opencv 4.11.0

### 1.4. Анализ базовой линии

**Низкий HOTA (4.73)** объясняется двумя факторами:

1. **Случайные REID-фичи**: оригинальная модель `mars-small128.pb` (TensorFlow) недоступна в нашем окружении. Вместо неё используются случайные 128-мерные векторы, что делает компонент ассоциации по внешности (cosine distance) неработоспособным. Это видно по почти нулевым значениям AssA (0.16–1.53).

2. **Оригинальный детектор POI**: детекции из `det.txt` предоставлены MOT Challenge и не являются оптимальными по современным стандартам. DetJ варьируется от 28.73 (MOT16-09) до 60.98 (PETS09-S2L1).

**Высокий FPS (334.57)** объясняется тем, что:
- Трекинг работает на предзагруженных детекциях (нет инференса детектора)
- REID-фичи предвычислены (нет инференса REID-модели)
- Остаётся только Kalman filter + linear assignment, что очень быстро

**Вывод**: базовая линия зафиксирована. Основное направление улучшения — интеграция современных детекторов и REID-моделей, что должно значительно повысить HOTA (особенно AssA за счёт настоящих фичей) при сохранении FPS ≥ 5.

## 2. Интеграция детекторов

Подробный отчёт: `report/stage2_detectors.md`

### 2.1. Реализованные детекторы

Создан унифицированный интерфейс `detectors/base_detector.py` с фабрикой `create_detector(name)`. Интегрированы 4 модели (все через Ultralytics):

| Имя | Модель | mAP (COCO) | Params |
|-----|--------|------------|--------|
| `yolo11n` | YOLO11 nano | 39.5 | 2.6M |
| `yolo11s` | YOLO11 small | 47.0 | 9.4M |
| `yolov8n` | YOLOv8 nano | 37.3 | 3.2M |
| `yolov8s` | YOLOv8 small | 44.9 | 11.2M |

### 2.2. Результаты сравнения (CPU, conf=0.3, IoU=0.5)

| Детектор | Precision | Recall | F1 | FPS |
|----------|-----------|--------|----|-----|
| **yolo11s** | 0.858 | 0.641 | 0.734 | **8.77** |
| yolo11n | 0.858 | 0.641 | 0.734 | 8.57 |
| yolov8n | 0.858 | 0.641 | 0.734 | 8.41 |
| yolov8s | 0.858 | 0.641 | 0.734 | 5.76 |

### 2.3. Выбранный детектор: **YOLO11s**

Лучший по соотношению качество/скорость: F1=0.734, FPS=8.77 на CPU, поддержка сегментации (YOLO11-seg) для доп. задачи.
