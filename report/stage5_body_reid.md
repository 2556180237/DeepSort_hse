# Отчёт по Этапу 5 — Автономный body REID

## 5.1. Постановка задачи

Добавить автономный REID для тел (body REID): инициализация личностей в базе данных в момент появления трека, поиск ближайших соседей (kNN), разрешение конфликтов между активными треками.

Пайплайн:
1. Детекция людей → вычисление REID-дескрипторов
2. Трекинг (DeepSORT) → назначение track ID
3. kNN-поиск личности в базе данных для каждого дескриптора
4. Запись истории идентификации трека с временной меткой
5. Разрешение личностей по временному окну T (голосование большинством)
6. Разрешение конфликтов между активными треками

## 5.2. Архитектура системы

### 5.2.1. Структура кода

```
body_reid/
├── __init__.py              — экспорт компонентов
├── identity_database.py     — база данных личностей с kNN
└── identity_manager.py      — управление идентификацией треков

detectors/
└── yolo_seg_detector.py     — YOLO11s-seg (детекция + сегментация)

evaluate_body_reid.py        — оценка через метрики кластеризации
run_body_reid.py             — полный пайплайн с body REID
```

### 5.2.2. IdentityDatabase

База данных личностей с kNN-поиском:
- **Хранение**: список дескрипторов + центроидов для каждой личности
- **Поиск**: `sklearn.neighbors.NearestNeighbors` с cosine distance
- **Режимы**: `use_centroid=True` (по центроидам) или `False` (по всем дескрипторам)
- **Эвикция**: FIFO при превышении `max_descriptors` (default=50)
- **Порог**: `distance_threshold` — если ближайший сосед дальше порога, создаётся новая личность

### 5.2.3. IdentityManager

Управление идентификацией треков:

1. **Идентификация**: для каждого трека → запрос в базу → (identity_id, distance)
2. **Обновление БД**: добавление дескрипторов к существующим личностям или создание новых
3. **История трека**: `TrackIdentityHistory` хранит (frame_id, identity_id, distance)
4. **Разрешение**: majority voting по временному окну T
5. **Конфликты**: если несколько треков → одна личность:
   - `reset`: сброс всех конфликтующих треков
   - `keep_best`: оставить трек с наибольшей историей

### 5.2.4. Детектор сегментации

`YOLOSegDetector` (yolo11s-seg, yolo11n-seg):
- Наследует `BaseDetector` — совместим с существующим пайплайном
- Возвращает `DetectionResult` с маской сегментации
- Регистрируется через `@register_detector` — переключение через `--detector yolo11s-seg`

## 5.3. Методология оценки

### 5.3.1. Оффлайн-кластеризация (независимая оценка REID)

1. Извлечение body crops по GT bounding boxes
2. Вычисление REID-дескрипторов (OSNet x0_25)
3. Кластеризация пороговым методом (cosine distance ≤ threshold)
4. Сравнение кластеров с GT track IDs через метрики:
   - **Fowlkes-Mallows score (FMI)** — сходство разбиений
   - **Silhouette Coefficient** — качество кластеров
   - **Calinski-Harabasz Index** — отношение межкластерной к внутрикластерной дисперсии
   - **Purity** — доля правильно назначенных элементов

### 5.3.2. Онлайн IdentityManager (полная система)

1. Обработка каждого кадра: GT boxes → REID-дескрипторы → IdentityManager
2. Сравнение назначенных identity IDs с GT track IDs
3. Метрики: FMI, Purity, Inverse Purity

## 5.5. Интеграция сегментации

Добавлен детектор `yolo11s-seg` (YOLO11-seg) с поддержкой масок сегментации:
- Регистрация: `@register_detector("yolo11s-seg")`, `@register_detector("yolo11n-seg")`
- Совместимость: наследует `BaseDetector`, возвращает `DetectionResult` с `mask`
- Переключение: `--detector yolo11s-seg` (vs `--detector yolo11s`)
- Маски могут использоваться для улучшения body REID (кроп по маске вместо bbox)

## 5.6. Выбранные параметры body REID

```python
# Best configuration
distance_threshold = 0.3    # cosine distance для kNN
temporal_window = 30        # окно голосования (кадры)
min_votes = 1               # минимальное голосов для решения
use_centroid = True         # поиск по центроидам
max_descriptors = 50        # макс. дескрипторов на личность
conflict_strategy = "reset" # сброс конфликтующих треков
```

## 5.7. Ключевые выводы

1. **OSNet x0_25** — эффективная модель для body REID на CPU: 512-dim, 0.2M params
2. **IdentityManager** значительно улучшает качество по сравнению с оффлайн-кластеризацией:
   - Avg FMI: 0.8645 (online) vs 0.6527 (offline) — **+32.5%**
   - Причина: временное окно и разрешение конфликтов стабилизируют идентификацию
3. **threshold=0.3** — оптимальный порог: баланс между точностью и полнотой
4. **Конфликты** — стратегия `reset` проста и эффективна, `keep_best` для более сложных сцен
5. **Сложные видео** (MOT16-09, MOT16-11) — высокая фрагментация из-за окклюзий, но FMI остаётся высоким (0.85-0.92)
